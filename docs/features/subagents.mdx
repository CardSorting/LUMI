Subagents let LUMI schedule up to five focused lanes through a three-request, work-conserving pool. Each lane gets its own prompt, model client, and context window, then returns a detailed result to the parent. Retry backoff releases its execution slot so untouched work keeps moving.

<Tip>
  Subagents is an experimental feature. Behavior may change in future releases.
</Tip>

## How It Works

When LUMI uses the `use_subagents` tool, it queues independent lanes by downstream priority. Each one:

- Gets its own prompt describing what to investigate
- Runs with a separate context window and token budget
- Can read/search files, run commands, use configured skills and MCP, and edit files when the lane has mutating authority
- Cannot use the browser, perform web search, or spawn nested subagents
- Returns a result focused on the most relevant file paths for the main agent to read next

Subagent costs are tracked across every retry and rolled into the parent budget. Progress UI and checkpoint writes are coalesced off the execution path; the parent alone publishes the terminal artifact.

## Enabling Subagents

Subagents are disabled by default. To turn them on:

1. Open LUMI Settings (click the gear icon in the LUMI panel)
2. Go to **Features**
3. Under the **Agent** section, toggle **Subagents** on

This setting applies in VS Code via extension state (`StateManager`).

## Using Subagents

LUMI does not automatically decide to use subagents. You need to ask for them in your prompt. When the feature is enabled and you mention subagents (or describe a task that benefits from parallel exploration), LUMI will use the `use_subagents` tool.

Example prompts:

- "Use subagents to explore how authentication works and where the database models are defined"
- "Spin up subagents to investigate the API routes, the test setup, and the deployment config"
- "I'm new to this codebase. Use subagents to map out the main entry points, the routing layer, and the data access patterns"

Each subagent prompt should describe a focused scope. Use `[execution_mode:read_only]` for research, `[execution_mode:mutation]` or `[write_set:path]` for writes, and `[depends_on:0]` only for a real dependency on zero-based lane 0. LUMI prioritizes lanes that unblock the longest downstream path and synthesizes the results in the parent.
You can also run only one subagent when the task is small enough that parallel discovery would be unnecessary overhead.

## Auto-Approve Behavior

Read/audit/planning lanes follow the **Read project files** auto-approve permission and receive only local read, skill-loading, and diagnostic tools. A lane that declares mutation follows the **Edit files** permission and requires approval once at the parent boundary when edit auto-approval is off; inner I/O then runs without repeated prompts while tool allowlists, hooks, locks, budgets, and the merge gate remain enforced.

A mixed swarm must satisfy both read and edit auto-approval. If either permission is unavailable, LUMI requests one approval for the entire batch.

In [YOLO mode](/features/auto-approve#yolo-mode), subagents are always auto-approved.

If auto-approve is off, LUMI will ask for your approval before launching subagents, showing you the prompts it plans to send.

## What Subagents Can Do

Default lanes can use the following tools; named subagent configurations may narrow the allowlist:

| Tool | Purpose |
|------|---------|
| `read_file` | Read file contents |
| `list_files` | List directory contents |
| `search_files` | Regex search across files |
| `list_code_definition_names` | List top-level classes, functions, and methods |
| `execute_command` | Run scoped commands in an authorized mutation lane |
| `use_skill` | Load and activate skills |
| `write_to_file`, `replace_in_file` | Write within an authorized mutation lane |
| MCP tools | Use configured MCP tools and resources in an authorized mutation lane |

Subagents cannot use the browser, perform web searches, or spawn their own subagents.

<Note>
  Commands run in the background under the parent launch authority. Declare mutation for commands that may modify files or system state.
  Subagents also benefit from command pipelines and filters to narrow output quickly before reading files, for example `rg ... | sort | uniq`.
</Note>

## When to Use Subagents

Subagents work best when you need broad context from multiple areas of a codebase at once:

- **Onboarding to an unfamiliar project**: Ask subagents to map out the architecture, key entry points, and data flow in parallel.
- **Investigating cross-cutting concerns**: Have separate subagents trace authentication, logging, and error handling simultaneously.
- **Pre-edit research**: Before making changes, use subagents to gather context from related files so the main agent can make informed edits without burning through its context window.
- **Large codebases**: When reading many files sequentially would consume too much of the main agent's context, subagents let you explore broadly without that tradeoff.

For small, focused tasks where you already know which files to look at, subagents add unnecessary overhead. Just ask LUMI directly.

## Governed parallel swarms (mutation lanes)

When `use_subagents` runs **multiple lanes that edit files** in parallel, LUMI uses the **governed execution harness** — not just read-only research mode.

| Concern | Behavior |
|---------|----------|
| File writes | `LockAuthority` for `mutation` lanes; read lanes skip locks |
| Workspace kanban | Lanes use `[propose_patch:…]` — **never direct kanban writes** |
| Private progress | `[local_roadmap:progress_note:…]` on agent projection |
| Swarm success | `MergeGate` + patch reconciliation before seal |
| Operator view | Incident console shows accepted/rejected patches, commit status |

**Quick start for roadmap-aware swarms:**

```
[execution_mode:mutation] [roadmap_item:NOW-42]
[local_roadmap:progress_note:NOW-42:implementing feature]
[propose_patch:attach_evidence:NOW-42:evidence=tests/feature.test.ts|rationale=tests pass|confidence=0.9]
```

Full reference: [Roadmap projection quick reference](/governed-roadmap-projection-quickref) · [Governed subagent execution](/governed-subagent-execution) · [Runbook](/governed-execution-runbook).
