LUMI steers long-running work through a project **`ROADMAP.md`** at the workspace root. Roadmap **governance gates** integrate with the completion pipeline — they can block `attempt_completion` when steering is out of sync.

**Key behavior (2026):** Governance remediation is **internal and automatic** at `attempt_completion`. Agents do **not** need to call `roadmap(action='validate')`, MCP tools, or slash commands to clear routine gates. Under multi-agent environments, the roadmap graduates into a pressure-aware coordination substrate ensuring stable, contention-free concurrent execution.

Implementation: `src/services/roadmap/` · Completion integration: `RoadmapCompletionGate.ts` + `completionGatePipeline.ts` · Shared copy: `RoadmapAutoGovernance.ts`

---

## Auto-governance at completion

When the agent calls `attempt_completion`, LUMI runs **`remediateRoadmapGatesInternally()`** before deciding whether to block:

1. **Bootstrap autofill** — if template placeholder phrases remain and `lumi.roadmap.autoBootstrapFill` is enabled
2. **Schema validation** — if `validation_pending` or after autofill wrote changes
3. **Checkpoint date stamp** — if section 11 `**Date:**` is missing or unparsable (mechanical fix only)

All steps use `RoadmapService` directly — no agent tool round-trips.

```text
attempt_completion
  → remediateRoadmapGatesInternally (autofill → validate → optional date stamp)
  → re-evaluate gates
  → pass OR block with ROADMAP.md edit guidance
```

If gates still fail after auto-remediation, the agent receives:

- A human-readable gate list with **per-gate edit instructions**
- A machine-parseable `<roadmap_governance_recovery>` block (RFC 7807-style extension)
- Explicit guidance: **edit ROADMAP.md** — do not call `validate` or MCP for governance

Mid-task (before completion), agents should **continue working**. The system prompt and write hints say governance runs at `attempt_completion`, not on every ROADMAP edit.

**Preflight readiness** uses a **non-mutating dry-run** (`dryRun: true`): it previews planned remediation steps without writing `ROADMAP.md` or `roadmap-state.json`. Auto-clearable gates (e.g. `validation_pending` only) do not surface as **blockers** in the preflight brief — instead, a non-blocking `<advisory>` in `<completion_gate_readiness>` confirms governance will run at `attempt_completion` (including planned remediation steps). The readiness block also carries a `governance_policy` attribute when the roadmap stage is present. Hard gates (schema errors, missing ROADMAP.md) still appear as blocking issues.

**Mid-task `validation_pending`:** After ROADMAP.md edits, workspace state marks `validation_pending: true`. Status reads (`guide`, `cockpit`, write hints) **do not** auto-validate — agents continue working until `attempt_completion` runs internal remediation.

**Mid-task gate display:** When only auto-clearable gates are closed (`validation_pending`, bootstrap placeholders), steering surfaces show a single informational note — not **⛔ attempt_completion blocked** and not stacked ⚠️ warnings — so agents are not pushed into validate/cockpit loops. The environment block includes an explicit **governance policy** line.

**Validate / bootstrap preview:** `roadmap(action='validate')` and `apply_bootstrap_fill` (preview mode) are **diagnostics only** — responses use `continueTaskMidPass` as the primary next call; autofill writes run at `attempt_completion`.

**Gate recovery copy:** Closed gates use per-gate `edit:` instructions from `GATE_EDIT_INSTRUCTIONS` (explain-gate, completion recovery XML, gate catalog) — file-action-first, not tool loops.

<Note>
  Optional diagnostics remain available: `/roadmap cockpit`, `/roadmap explain-gate`, `/roadmap explain-stale`, `roadmap(action='doctor')`. Use these for visibility — not as a required validate loop before every completion attempt.
</Note>

## Multi-Agent Coordination & Leases

To support concurrent execution across multiple collaborative agents without write contention, planning overlap, or state corruption, the runtime implements **Execution Ownership Leases**:

- **Mutation Lock Envelopes:** Agents must acquire a temporary exclusive lease for a specific task node or checkpoint before executing writes or modifying states.
- **Expiration Policies:** Leases are acquired via `acquireOrchestrationLease(workspace, agentId, taskId, durationSeconds)` and expire automatically (default: 5 minutes) to avoid execution starvation. Same-agent lease renewals are permitted.
- **Handoff Semantics:** Active leases are cleanly released using `releaseOrchestrationLease(workspace, agentId, taskId)` upon task completion or execution continuation.

---

## Pressure-Aware Scheduling & Cooldowns

Under production pressure, the runtime actively prevents planning storms, excessive validation cycles, and scheduler congestion by calculating an **Orchestration Pressure Score** (0.0 to 1.0) and running an **Admission Control Queue**:

- **Pressure Score Factors:** Dynamically calculated based on the count of active unexpired leases (0.2 per lock), the size of the active task queue (Now list) (0.05 per task), and the mutation rate (lineage edits in the last 5 minutes, 0.1 per mutation).
- **Admission Queue Backoffs:** The `scheduleAdmission(workspace, agentId, operation)` wrapper paces agent requests. If the pressure score is high (>= 0.8) or if a cooldown window is active, it blocks immediate execution (`admitted: false`) and returns exponential backoff parameters (up to 10 seconds), preventing runaway planning chains.
- **Lazy Validation Caching:** To minimize context over-hydrations and redundant CPU/IO execution cost, `validateRoadmap` caches validation results. If a request arrives within 5 seconds of a previous validation and the `ROADMAP.md` hash is unchanged, the heavy parsing pipeline is bypassed.

---

## Concurrent Continuation & Version Vectors

When agents run recovery cycles, asynchronous tasks, or retry loops, the runtime prevents split-brain recovery races and fork divergence through **Freshness Validation**:

- **Monotonic Version Vectors:** Every task node mutation, lease modification, and continuation anchor registration increment a sequence counter inside the state's `version_vectors` dictionary.
- **Divergence Detection:** Before resuming from a continuation anchor, the executor verifies version vector freshness using `verifyAnchorFreshness(workspace, key, expectedVersion)`. If the current sequence counter diverges from the expected version, the runtime rejects resumption and requires coordination.

---

## Scoped Cognitive Partitioning

As orchestration scales, the roadmap runtime shape-shifts agent views into local operational contexts, preventing cognitive overload and prompt bloat:

- **Local Reasoning Corridors:** Cockpit snapshots (`buildCockpitPayload`) and steering blocks (`formatRoadmapSteeringBlock`) automatically filter out locks held by other agents and tasks locked by others, localizing the focus-scoped execution graph to this agent's immediate task neighborhood.
- **Steering Lineage Filtering:** Lineage ledger histories filter out entries generated by other agents.
- **Verbose Escape Hatch:** Agents can request unfiltered context views by supplying the `--verbose` flag.

## VS Code settings

All settings use the `lumi.roadmap.*` prefix in root `package.json`:

| Setting | Default | Purpose |
|---------|---------|---------|
| `lumi.roadmap.enabled` | `true` | Master switch for roadmap steering |
| `lumi.roadmap.autoBootstrap` | `true` | Create `ROADMAP.md` from workspace evidence when missing |
| `lumi.roadmap.autoBootstrapFill` | `true` | Apply evidence autofill (also runs at completion) |
| `lumi.roadmap.blockKanbanOnValidationPending` | `true` | Block completion when ROADMAP changed since last validate (auto-cleared at completion) |
| `lumi.roadmap.blockKanbanOnBootstrapIncomplete` | `true` | Block when bootstrap template phrases remain (autofill runs first) |
| `lumi.roadmap.warnOnStaleBeforeComplete` | `true` | Block when checkpoint is stale vs git activity |
| `lumi.roadmap.failClosedCompletionGates` | `true` | Block completion when roadmap gate evaluation throws |

Environment overrides: `MIRA_ROADMAP_*` / `DIETCODE_ROADMAP_*` (see `RoadmapConfig.ts`).

## Gate catalog

Gates are defined in `RoadmapGateCatalog.ts`. Blocking gates depend on settings above.

| Gate | Typical auto-fix at completion | If still blocked |
|------|-------------------------------|------------------|
| `roadmap_present` | Session auto-bootstrap | Create ROADMAP.md at workspace root |
| `bootstrap_complete` | Auto-autofill | Edit remaining template phrases |
| `validation_current` | Auto-validate | Repair schema in ROADMAP.md |
| `checkpoint_fresh` | Date stamp (missing/invalid date only) | Update section 11 narrative + date |
| `schema_valid` | — | Repair all 12 required sections |

Stale checkpoint due to **git activity** or **expired age** is not auto-rewritten — the agent must update section 11 content.

## File watcher

`extension.ts` registers `RoadmapFileWatcher` so external edits to `ROADMAP.md` set `validation_pending` in workspace state. Auto-validation clears this at completion — agents do not need a manual validate step.
