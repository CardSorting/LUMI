Auto Approve lets you decide which actions LUMI can take without prompting you each time. It keeps you out of approval popups during routine work while letting you keep tight control over high-risk actions.

If you find yourself repeatedly clicking approve for the same safe operations, Auto Approve fixes that.

## How It Works

Auto Approve is evaluated per invocation by `ExecutionFunnel`. The registered handler first declares a pure intent; the funnel then evaluates every declared capability against current settings and execution policy. It records one automatic or explicit decision and issues a permit only after approval resolves.

A few details matter:

- **Workspace vs outside workspace**: "Read all files" and "Edit all files" only extend the base toggle. If the base toggle is off, the "all files" option does nothing.
- **Terminal commands**: Central command classification, command-permission policy, trusted-command policy, and the current settings all apply. A model-supplied flag cannot make an unknown command safe.
- **Notifications**: If enabled, LUMI sends OS-level notifications when approval is required, and when an auto-approved terminal command has been running for 30 seconds.

## Permissions

| Setting | What It Allows |
|---------|----------------|
| Read project files | Read files, list files, search in your workspace |
| Read all files | Read files outside your workspace (requires base toggle) |
| Edit project files | Create and edit files in your workspace |
| Edit all files | Edit files outside your workspace (requires base toggle) |
| Execute safe commands | Run terminal commands marked safe |
| Execute all commands | Run commands requiring approval (requires base toggle) |
| Use the browser | Browser tool for web fetching and searching |
| Use MCP servers | MCP tools and resources |
| Enable notifications | Notifies you about long-running commands |

<Warning>
"Read all files" and "Edit all files" only matter if their base toggle is enabled. They extend access outside your workspace.
</Warning>

## Safe vs Approval-Required Commands

LUMI classifies the exact normalized command centrally. Read-only and verification tiers can use **Execute safe commands**; trusted commands and **Execute all commands** follow their respective policies. These are examples, not guarantees.

**Commonly treated as safe:**
- `npm run build`, `npm test` - Build/test output
- `git status`, `ls -la`, `cat package.json` - Read-only commands

**Commonly requires approval:**
- `npm install <pkg>` - Modifies dependencies
- `rm -rf <path>` - Deletes files
- `mv <a> <b>` - Moves files (can overwrite)
- `sed -i ...` - In-place file edits

## Recommendations

A good default setup:
- Enable **Read project files**
- Leave edits, commands, browser, and MCP off until you have a specific reason

If you enable edits, use [Checkpoints](/core-workflows/checkpoints) so you can roll back quickly.

## YOLO Mode

YOLO mode changes agent interaction and mode-transition behavior. It is not a compatibility override for execution approval: `ExecutionFunnel` still evaluates the current per-capability settings and policy, records one decision, and issues a linked permit.

<Note>
**Warning:** YOLO mode reduces interactive guidance and follow-up questions. Configure Auto Approve permissions separately; fencing, command policy, hooks, replay protection, and the approval-linked permit remain active.
</Note>

### What Gets Auto-Approved

When YOLO mode is enabled, the agent may proceed with best judgment rather than asking task-clarification questions, and mode transitions can be less interactive. Automatic execution still depends on the corresponding Auto Approve setting and execution policy.

### When to Use YOLO Mode

**Rapid prototyping** where you want zero friction and don't care about potential mistakes. Perfect for throwaway experiments.

**Trusted, repetitive tasks** where you've validated LUMI's approach and want to eliminate approval overhead.

**Demonstration purposes** where you want to show LUMI's capabilities without interruptions.

### What Could Go Wrong

LUMI could:
- Delete important files without warning
- Execute commands that modify system settings
- Make network requests to external services
- Overwrite configuration files
- Install or uninstall packages
- Commit and push changes to version control

### Best Practices for YOLO Mode

- **Start with isolated environments.** Use throwaway projects or sandboxed environments first.
- **Be specific with requests.** Vague instructions + unlimited permissions = unpredictable results.
- **Monitor the output.** LUMI still shows what it's doing. Watch the terminal and file changes.
- **Keep version control handy.** Git becomes your safety net.

### Enabling YOLO Mode

Navigate to LUMI Settings, then Features, and check "YOLO Mode." Configure Auto Approve permissions independently for the capabilities you intend to admit automatically. Uncheck YOLO mode to restore normal interactive guidance.

## Implementation (code paths)

| Setting | Storage key | Effect |
|---------|-------------|--------|
| Per-category auto-approve | `autoApprovalSettings` | Allows `ExecutionFunnel` to record automatic approval when the complete intent and policy are eligible |
| YOLO Mode | `yoloModeToggled` (global state) | Changes prompt/follow-up and mode-transition behavior; does not override execution admission |

**Code:**

- Settings persistence: `src/core/controller/state/updateSettings.ts`
- YOLO prompt behavior: `src/core/prompts/system-prompt/` variants (e.g. `next-gen/template.ts`)
- Sole approval/permit gate: `src/core/task/tools/execution/ExecutionFunnel.ts`
- Pure handler contract: `src/core/task/tools/types/ToolContracts.ts`
- MCP per-tool setting: evaluated by `ExecutionFunnel` from `mcpHub.connections`

YOLO Mode is the highest-risk setting. Use [checkpoints](/core-workflows/checkpoints) as your rollback layer when experimenting.
