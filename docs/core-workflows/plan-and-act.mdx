Plan and Act are internal phases that separate thinking from doing. You do not switch modes manually — DietCode manages the transition for you.

<Tip>
	**New to Plan & Act?** Watch [Plan & Act Deep Dive](https://youtu.be/b7o6URFPp64) for the concepts. The product now automates the handoff between phases.
</Tip>

## How It Works

Every new task follows this flow automatically:

1. **Plan phase** — DietCode explores your codebase (read-only), asks clarifying questions if needed, and architects an approach
2. **Plan delivery** — The finalized plan appears in chat as a structured summary (non-blocking — you can keep typing)
3. **Act phase** — DietCode automatically transitions to implementation and begins making changes (with your approval on each tool use)

There is no Plan/Act toggle in the chat UI. Mode transitions are invisible by design — similar to how Cursor, Devin, and other agents handle plan-then-execute workflows.

<Note>
	**YOLO mode** skips the planning phase and starts directly in Act mode for maximum autonomy. Enable it in Settings when you want immediate implementation without a planning step.
</Note>

## Plan Phase

During planning, DietCode can read your codebase, run searches, and discuss strategy, but cannot modify files or execute commands.

This constraint keeps the conversation focused on understanding before changes. Use this phase to:

- Explore unfamiliar codebases before making changes
- Discuss architecture decisions and tradeoffs
- Identify edge cases and potential issues upfront
- Create a clear implementation strategy

## Act Phase

After the plan is delivered, DietCode retains full context from planning and implements the solution. File edits and commands still require your approval (unless auto-approve is configured).

During Act Mode, send follow-up messages at any time to steer direction — even while the agent is working. DietCode incorporates feedback immediately and may automatically return to planning when you request a scope pivot (e.g., "let's replan with a different approach").

## Typical Workflow

1. Describe what you want to build
2. DietCode explores relevant files and understands the codebase
3. DietCode presents a plan in chat
4. Implementation begins automatically — approve tool uses as they appear
5. DietCode completes the task with `attempt_completion`

For complex projects, you may iterate: send feedback mid-task, ask for scope changes, or start a new task with `/newtask` context.

## When Planning Helps Most

| Scenario | Why plan first |
|----------|----------------|
| Starting new features where the approach isn't obvious | Reduces rework |
| Debugging tricky issues | Investigation before edits |
| Architectural decisions affecting multiple files | Coherent design |
| Understanding complex workflows before modifying | Safer changes |
| Learning a new codebase | Read-only exploration |

For obvious quick fixes (typos, missing imports), enable **YOLO mode** to skip planning and implement immediately.

## Using Different Models for Each Phase

You can configure separate models for Plan and Act phases in Settings. This is useful when you want a stronger reasoning model for planning and a faster model for implementation.

1. Open DietCode Settings
2. Enable "Use different models for Plan and Act modes"
3. Select your preferred model for each phase

The agent automatically uses the correct model when transitioning between phases.

**Example configurations:**

| Use Case | Plan Phase | Act Phase |
|----------|------------|-----------|
| Cost optimization | GLM 4.6 | Grok Code Fast |
| Maximum quality | Claude Opus | Claude Sonnet |
| Speed-focused | Gemini 3 Flash | Cerebras |

## Using `/deep-planning`

For complex tasks that need thorough analysis, use the `/deep-planning` slash command. This triggers an extended planning session where DietCode:

1. Explores the codebase systematically
2. Identifies all affected files and dependencies
3. Creates a detailed implementation plan document
4. Presents the plan and automatically transitions to implementation

See the [Deep Planning docs](/features/deep-planning) for more details.

## Choosing the Right Approach by Task Size

### Small tasks: YOLO / Act only

For quick fixes like typos, simple bug fixes, or following established patterns, enable YOLO mode or expect a brief planning pass before rapid implementation.

**Examples:** Fix a typo, add a missing import, update a config value, rename a variable.

### Medium tasks: Automatic Plan → Act

For most development work, DietCode plans first, then implements. This is the default for features that touch a few files and have some complexity.

**Examples:** Add a new API endpoint, implement a UI component, fix a bug that requires investigation, refactor a single module.

### Large tasks: Use `/deep-planning`

For complex features spanning multiple files, requiring architectural decisions, or taking multiple sessions, use `/deep-planning` for a documented implementation spec.

**Examples:** Add a new feature across frontend and backend, major refactoring, implementing a new system or integration, multi-step migrations.

## Tips

- Plans appear as structured chat messages — scroll up to review anytime during implementation
- Use [file mentions](/core-workflows/working-with-files) to point DietCode at relevant files
- Enable [Checkpoints](/core-workflows/checkpoints) before large implementations so you can roll back
- [Focus Chain](/features/focus-chain) tracks implementation progress after the plan phase
- Send follow-up messages to steer direction — no mode switching required
