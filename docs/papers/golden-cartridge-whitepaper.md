# Golden Cartridge: Technical Whitepaper

*A technical overview of LUMI's resource-constrained tool composition and cached evidence facade.*

> **Related:** [Golden Cartridge Brief](golden-cartridge-brief.md) · [Golden Cartridge Philosophy](golden-cartridge-philosophy.md) · [Tools Reference](../tools-reference/golden-cartridge.mdx) · [README](README.md)

---

## 1. Introduction & Objectives

The **Golden Cartridge Protocol** introduces a resource-constrained developer agent methodology. Its primary objective is to manage the mass—both in context size and codebase modifications—of agent activities.

### Technical Scope & Boundaries
* **Overlay Architecture**: The protocol is implemented as a one-way facade tool (`golden_cartridge`). It coordinates but never replaces the core tool schema.
* **Non-Enforcing**: The facade does not override repository write rules, block execution, or authorize task completion. All mutating operations retain their default user-approval gates.
* **Telemetry and Cache Acceleration**: Utilizes workspace revision tracking and mutation signals to cache intermediate structural evidence, reducing redundant read operations.

## 2. Facade Architecture & Tool Coordination

The class `GoldenCartridgeToolHandler` (implemented in [GoldenCartridgeToolHandler.ts](file:///Users/bozoegg/Downloads/codemarie-new/src/core/task/tools/handlers/GoldenCartridgeToolHandler.ts)) acts as the single entry point. It receives a `verb` and an optional JSON-encoded `payload`, then dispatches commands to delegated core handlers.

```
                  ┌───────────────────────────────┐
                  │   GoldenCartridgeToolHandler  │
                  └───────────────┬───────────────┘
                                  │
         ┌────────────────────────┼────────────────────────┐
         │ (delegation)           │ (delegation)           │ (delegation)
         ▼                        ▼                        ▼
┌─────────────────┐      ┌─────────────────┐      ┌─────────────────┐
│   ProjectMap    │      │   SearchFiles   │      │   ApplyPatch    │
│   (for trace)   │      │   (for slice)   │      │(for patch_smallest)
└─────────────────┘      └─────────────────┘      └─────────────────┘
```

The handler maps its verbs to the following core subsystems:
1. **Structural Mapping (`trace`, `resolve_authority`)**: Delegates to `ProjectMapHandler` and ranks results.
2. **Lexical Context (`slice`, `find_reuse`)**: Composes `SearchFilesToolHandler` and `ListCodeDefinitionNamesToolHandler` output.
3. **Mutation (`patch_smallest`)**: delegates to the core `ApplyPatchHandler`.
4. **Validation (`disprove`)**: Composes unit test file discovery with command execution via `ExecuteCommandToolHandler`.
5. **Memory & Context (`compress`)**: Triggers `CognitiveMemorySnapshotHandler` and `CondenseHandler`.
6. **Analysis (`compare_mass`, `design_compact`, `measure`, `reclaim`, `seal`)**: Operates directly on `TaskState` metrics and mathematical candidate comparisons.

## 3. Caching & Invalidation Mechanics

To prevent redundant repository scans and shell executions, the handler implements two layers of caching:

### 3.1. Evidence Caching
For discovery verbs (`trace`, `slice`, `resolve_authority`, `find_reuse`), outputs are stored in `TaskState.goldenCartridgeEvidenceCache`.
* **Cache Key**: A stable string generated by sorting payload attributes alphabetically (excluding transient tags like `refresh` or `bypassCache`).
* **Invalidation Trigger**: Checked against `TaskState.goldenCartridgeEvidenceGeneration`. Any file modification (`didEditFile`) or workspace revision change increments this generation and clears the cache.

### 3.2. Validation Caching
The results of unit tests and validation checks run via `disprove` are cached in `TaskState.goldenCartridgeValidationHistory`.
* **Revision Locking**: Validation outcomes are tagged with the active repository revision (`repositoryIdentity`), which combines `goldenCartridgeCanonicalWorkspaceRevision` (from `NativeMutationManager.workspaceRevision`) and the task-local `goldenCartridgeEvidenceGeneration`.
* **Reusability Check**: If `disprove` is called for a matching command, and the repository revision has not changed, the passing status is reused, preventing redundant test execution.

## 4. Candidate Tradeoff Analysis

The `compare_mass` verb performs a deterministic multidimensional tradeoff comparison of code modifications using the `deterministicMassComparison` algorithm.

### 4.1. The Mass Matrix
Each candidate solution is graded on structural parameters defined in `SolutionCandidate`:
* **Files Touched** (lower is better)
* **Public Interfaces Added** (lower is better)
* **Dependencies Introduced** (lower is better)
* **New Abstractions/Authorities** (lower is better)
* **Uncertainty / Risk** (lower is better)
* **Confidence / Correctness** (higher is better)

### 4.2. Pareto Dominance Sorting
The algorithm evaluates candidates based on mathematical dominance:
* Candidate $A$ dominates candidate $B$ if it is at least as good as $B$ on all metrics and strictly better on at least one.
* If a single candidate dominates all others, it is recommended as the `lowestMassCandidate`.
* If multiple undominated candidates exist, they are listed as active tradeoffs for the user to select.

## 5. Handoff & The Seal Receipt

At the end of a task, the agent calls the `seal` verb. This generates an informational cartridge receipt summarizing:
* **The acceptance requirement** defined in the working set.
* **The critical path** and file surfaces modified.
* **The validation basis** (the checklist of passing checks).
* **Missing evidence** and remaining risks or unresolved ambiguities.
* **Completion confidence** (`high | medium | low`).

This receipt is formatted as a structured JSON object. It serves as clean metadata for subsequent subagents or human operators, assuring that all modifications are backed by grounding evidence.
