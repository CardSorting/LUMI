DietCode uses some specialized terms to describe its internal systems. This guide translates those technical concepts into plain English.

## System Components

| Current Term | Former Term (Deprecated) | Description |
| :--- | :--- | :--- |
| **Code Analysis Engine** | Spider Engine | The system that allows DietCode to "see" and understand the structure of your files. |
| **Context Memory** | BroccoliDB | A local SQLite database that stores what DietCode has learned about your project during a task. |
| **Connection Layer** | HostBridge | Connects the AI "brain" to your editor (VS Code, JetBrains, etc.). |
| **Communication Protocol** | Protobus | The high-performance way different parts of DietCode talk to each other. |
| **Sub-agents** | Subagents | Specialized assistants that DietCode can create to help with specific sub-tasks. |
| **Technical Debt** | Pathogen | A file or module that is too complex, messy, or has too many bugs. |
| **Architecture Standards** | JoyZoning | Rules that ensure code is placed in the right directory and follows best practices. |
| **Resource Usage** | Metabolic Pressure | How much memory (RAM) or processing power the system is using. |
| **Automated Diagnostics** | Forensic Healing | When DietCode detects a structural error and fixes it automatically. |
| **Core Rule** | Axiom | A fundamental rule about how your project's code should be organized. |
| **Impact Analysis** | Blast Radius | Figuring out which other files might break if you change a piece of code. |
| **Context Grounding** | Grounding | Ensuring the AI bases its decisions on real code it has actually read. |
