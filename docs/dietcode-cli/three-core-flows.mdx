---
title: "Headless Mode"
description: "Run DietCode autonomously in scripts, CI/CD pipelines, and automated workflows"
---

Headless mode runs DietCode without an interactive interface — perfect for automation, scripting, and CI/CD pipelines where human interaction isn't possible or desired. DietCode executes tasks, produces clean text or JSON output, and exits when complete.

For collaborative, conversational development, see [Interactive Mode](/dietcode-cli/interactive-mode) instead.

<Note>
**Migrating from an older CLI version?** Instance commands (`dietcode instance new/list/kill`) have been removed in DietCode CLI 2.0. The new architecture is simpler — just use `dietcode -y "task"` for headless execution.
</Note>

## When Headless Mode Activates

DietCode automatically enters headless mode when any of these conditions are met:

| Invocation | Reason |
|------------|--------|
| `dietcode -y "task"` | `-y`/`--yolo` flag forces headless |
| `dietcode --json "task"` | `--json` flag forces headless |
| `cat file \| dietcode "task"` | stdin is piped |
| `dietcode "task" > output.txt` | stdout is redirected |

If none of these apply (e.g., running `dietcode` or `dietcode "task"` in a terminal), DietCode launches in [interactive mode](/dietcode-cli/interactive-mode).

## YOLO Mode (Fully Autonomous)

The `-y` or `--yolo` flag enables fully autonomous operation — DietCode approves all actions and runs without prompts:

```bash
dietcode -y "Run the test suite and fix any failures"
```

In YOLO mode:
- All actions are auto-approved
- Output is plain text (non-interactive)
- Process exits automatically when complete
- Perfect for CI/CD and scripts

<Warning>
YOLO mode gives DietCode full autonomy. Run on a clean git branch so you can easily revert changes if needed.
</Warning>

### Mode Selection

Control whether DietCode plans first or acts immediately:

```bash
# Start in Plan mode (analyze before acting)
dietcode -y -p "Design a REST API for user management"

# Start in Act mode (default)
dietcode -y -a "Fix the typo in README.md"
```

## Piping Context

Pipe file contents or command output into DietCode to provide context:

```bash
# Explain a file
cat README.md | dietcode "Summarize this document"

# Review git changes
git diff | dietcode "Review these changes and suggest improvements"

# Analyze command output
npm test 2>&1 | dietcode "Analyze these test failures and fix them"

# Pipe a GitHub PR diff
gh pr diff 123 | dietcode -y "Review this PR"
```

When stdin is piped, DietCode automatically enters headless mode — the piped content becomes part of the task context.

## Chaining Commands

Pipe DietCode's output into another DietCode instance for multi-step workflows:

```bash
# Explain changes, then write a commit message
git diff | dietcode -y "explain these changes" | dietcode -y "write a commit message for this"

# Generate code, then write tests
dietcode -y "create a fibonacci function" | dietcode -y "write unit tests for this code"

# Fun: Generate a poem about your code
git diff | dietcode -y "explain" | dietcode -y "write a haiku about this"
```

## JSON Output

Use `--json` for machine-readable output that's easy to parse in scripts:

```bash
dietcode --json "List all TODO comments in the codebase" | jq '.text'
```

JSON output follows the same format as task files in `~/.dietcode/data/tasks/<id>/ui_messages.json`.

**JSON Message Schema:**

| Field | Type | Description |
|-------|------|-------------|
| `type` | `"ask"` or `"say"` | Message category |
| `text` | `string` | Message content |
| `ts` | `number` | Unix timestamp (ms) |
| `reasoning` | `string` | (Optional) Model reasoning |
| `partial` | `boolean` | (Optional) Streaming flag |

## Including Images

Attach images to your headless task:

```bash
dietcode -y -i screenshot.png "Fix the layout issue shown in this screenshot"

# Or reference inline
dietcode -y "Fix the UI shown in @./design-mockup.png"
```

## Timeout Control

Set a maximum execution time to prevent runaway tasks:

```bash
dietcode -y --timeout 600 "Run full test suite"
```

## Environment Variables

Control DietCode behavior via environment variables — useful for CI/CD where you can't use interactive configuration.

**CLINE_DIR** — Custom configuration directory:
```bash
export CLINE_DIR=/path/to/config
dietcode -y "your task"
```

**CLINE_COMMAND_PERMISSIONS** — Restrict allowed commands:
```bash
export CLINE_COMMAND_PERMISSIONS='{"allow": ["npm *", "git *"], "deny": ["rm -rf *"]}'
dietcode -y "your task"
```

See [Configuration](/dietcode-cli/configuration#environment-variables) for full documentation.

## CI/CD Integration

### GitHub Actions Example

Automate PR reviews with DietCode:

```yaml
name: AI Code Review

on:
  pull_request:
    types: [opened, synchronize]

jobs:
  review:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
        with:
          fetch-depth: 0
      
      - uses: actions/setup-node@v4
        with:
          node-version: '22'
      
      - name: Install DietCode
        run: npm install -g dietcode
      
      - name: Configure DietCode
        run: dietcode auth -p anthropic -k ${{ secrets.ANTHROPIC_API_KEY }}
      
      - name: Review PR
        run: |
          git diff origin/main...HEAD | dietcode -y "Review this PR for:
          - Potential bugs
          - Security issues  
          - Performance concerns
          - Code style violations
          
          Provide a summary of findings."
```

### Shell Script Example

Create a reusable code review script:

```bash
#!/bin/bash
# review.sh - AI-powered code review

set -e

# Get the diff
DIFF=$(git diff HEAD~1)

if [ -z "$DIFF" ]; then
  echo "No changes to review"
  exit 0
fi

# Run DietCode review
echo "$DIFF" | dietcode -y --json "Review this code diff for issues" | jq -r '.text'
```

## Common Use Cases

| Use Case | Example |
|----------|---------|
| Code review | `git diff \| dietcode -y "Review these changes"` |
| Fix test failures | `dietcode -y "Run tests and fix any failures"` |
| Generate release notes | `git log --oneline v1.0..v1.1 \| dietcode -y "Write release notes"` |
| Fix lint errors | `dietcode -y "Fix all ESLint errors in src/"` |
| Update dependencies | `dietcode -y "Update dependencies with known vulnerabilities"` |
| Migrate code patterns | `dietcode -y "Update all deprecated React lifecycle methods"` |
| PR automation | `gh pr diff 123 \| dietcode -y "Review this PR"` |
| Batch processing | `dietcode -y --json "List all TODO comments" \| jq '.text'` |
